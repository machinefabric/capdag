Here are the twenty points summarizing the task integration in MachFab:

1. **Task System Overview**: The task system in MachFab is a wrapper around machine execution, resolving input files, building or receiving an execution plan, running the DAG through the capdag infrastructure, tracking progress, and storing results.

2. **Trigger Fuses**: Tasks can be created via `TriggerFuse` events, which trigger specific machines (e.g., `ExecuteMachine`) and control flags that allow the UI to cancel a running task by setting `aborted = true`.

3. **Cap Interpretation**: The `CapInterpreterOp` bridges the task system with the DAG executor, allowing for file resolution, progress chain, and automation tests.

4. **File Resolution**: Input files pass through several stages before reaching a cartridge: source listings, copying files to the container directory, and passing copied paths as input data for the first step's `InputSlot` nodes.

5. **Progress Chain**: The full progress chain from cartridge to UI includes logging, progress updating, control flag checking, and polling of task state in the database.

6. **Activity and Idle Detection**: The UI distinguishes "running" from "idle" based on frame activity, keeping alive tasks that don't change their progress value over time (keepalive frames).

7. **Task Completion**: When DAG execution finishes, results are stored in the task record, progress is set to 1.0, and task state → `Completed`.

8. **Automation Tests**: Automation test scripts exercise the full task pipeline, creating test files, triggering tasks via the automation API, waiting for task completion, verifying results, and testing timeouts.

9. **Timeouts vs Activity**: If a task stays "running" indefinitely due to the stderr blocking bug, the automation test hangs rather than failing cleanly, while keepalive frames prevent the engine's 120-second timeout but don't guarantee the test will complete.

10. **Keepalive Frames**: Keepalive frames maintain the engine alive even when not actively processing tasks, allowing for continuous execution of tasks without explicit user action (e.g., dragging a file onto a capability in the UI).

11. **Cartridge Security Bookmarks**: macOS runtime security bookmarks are used to sandbox access to files on the cartridge container directory, ensuring that only authorized processes can read or write data.

12. **DAG Execution**: DAG execution is handled by the `MachineExecutor` class, which calls the resolved plan and inputs with the task system.

13. **Task Parameters**: The task system uses a values map to pass task parameters (input files, slot settings) to the planner's slot resolution during planning phases.

14. **Cap Settings**: Cap settings are passed from the task context to the planner as well, allowing for flexible configuration of cap-related tasks.

15. **Progress Mapping**: The `map_progress()` function maps the DAG's [0.0, 1.0] range into the task's allocated step range, enabling progress tracking and reporting in the UI.

16. **DAG Resolution**: The `cap_interpreter` class resolves input files using the `FileOperationCoordinator`, which copies files to the cartridge container directory accessible to processes without additional entitlements.

17. **Cartridge Path Resolution**: Input files pass through several stages before reaching a cartridge: source listings, copying files to the container directory, and passing copied paths as input data for the first step's `InputSlot` nodes.

18. **Task State Updates**: Task state is updated periodically during execution by polling the database and updating the task record in SQLite.

19. **Error Handling**: Tasks can be aborted or failed due to various reasons (e.g., timeout, cartridge error, missing data), triggering the `ExecutionError` variant of the success or failure messages displayed in the UI.

20. **Automation API Integration**: Automation test scripts interact with the task system via the automation API, creating test files, triggering tasks, waiting for completion, verifying results, and testing timeouts.
